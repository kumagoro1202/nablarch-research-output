# shogun-multi-perspective-evaluation-integrator

## What This Skill Does

複数のステークホルダー視点から作成された評価レポート群を入力として受け取り、以下を出力する包括的分析スキル：

1. **横断分析テーブル**: 視点×評価軸のマトリクスで全体像を可視化
2. **リリースブロッカー**: 複数視点から指摘された致命的課題の特定
3. **優先度付き改善提案**: P0（ブロッカー）〜P3（中長期）の優先度付けと工数概算
4. **トレードオフ分析**: 相反する要件・制約の関係性マッピング
5. **殿の判断事項リスト**: 家老/将軍だけでは決められない意思決定ポイントの明確化

## When to Use

以下のような状況で使用する：

- **複数の視点別評価レポートが揃った時**: A.管理者、B.スタッフ、C.開発者、D.運用、E.経営、F.セキュリティ、G.法務など、3つ以上の視点別評価が完成した段階
- **リリース判定が必要な時**: 「このプロジェクトはリリース可能か？」を判断するために、複数視点からの懸念を統合して評価する
- **優先順位付けが必要な時**: 限られたリソースで「何を優先して改善すべきか」を決定する
- **トレードオフの可視化が必要な時**: セキュリティ vs ユーザビリティ、スピード vs 品質など、相反する要件を整理する
- **意思決定事項を抽出したい時**: 経営層や上様（殿）に判断を仰ぐべき事項を明確にする

このスキルは、**評価段階の最終工程**である。個別の視点別評価が完了した後に使用する。

## Workflow

### Phase 1: 入力準備

1. **入力パラメータの確認**:
   - `report_files`: 視点別評価レポートのファイルパスリスト（Markdown形式）
   - `project_name`: 評価対象プロジェクトの名前
   - `target_version`: 評価対象のバージョン（例: v1.5.0）
   - `output_path`: 統合レポートの出力先パス

2. **入力ファイルの検証**:
   - 各レポートファイルが存在することを確認
   - Markdown形式であることを確認
   - 最低3つ以上のレポートが必要（2つ以下では横断分析の意味が薄い）

3. **視点の識別**:
   - 各レポートから視点名を抽出（例: "A. 管理者", "B. スタッフ", "C. 開発者"）
   - 視点の略称を生成（例: A, B, C）

### Phase 2: 全レポート読み込み・構造解析

1. **各レポートを読み込む** (Read tool使用)

2. **構造解析**:
   各レポートから以下のセクションを抽出:
   - **良い点**: 高評価された項目
   - **不足点・問題点**: 指摘された課題
   - **改善提案**: 提案された改善案（優先度・工数を含む）
   - **総合評価・スコア**: 総合評価（A/B/C評価、点数、星評価など）
   - **リスク**: 法令違反リスク、セキュリティリスク、経営リスクなど

3. **データ構造化**:
   各視点のデータを以下の形式で整理:
   ```
   {
     "視点名": "A. 管理者",
     "略称": "A",
     "総合スコア": "B+",
     "良い点": ["...", "..."],
     "課題": ["...", "..."],
     "改善提案": [
       {"提案": "...", "優先度": "最優先", "工数": "3人日"},
       ...
     ],
     "リスク": ["...", "..."]
   }
   ```

### Phase 3: 横断分析

1. **評価軸の特定**:
   全視点から共通の評価軸を抽出:
   - UI/UX
   - 機能充足度
   - 法令遵守
   - データ安全性
   - 保守性
   - 拡張性
   - パフォーマンス
   - コスト
   - etc.

2. **横断分析テーブルの作成**:
   視点（列）× 評価軸（行）のマトリクスを作成:

   | 評価軸 | A.管理者 | B.スタッフ | C.開発者 | D.運用 | E.経営 | F.セキュリティ | G.法務 |
   |--------|---------|-----------|---------|--------|--------|-------------|--------|
   | 総合スコア | B+ | 0/100 | B+ | ★3.0/5 | 推奨 | 中リスク | 条件付適合 |
   | UI/UX | ✅ 直感的 | ❌ 機能なし | - | ✅ トレイ統合 | ✅ 対応 | - | - |
   | 機能充足度 | ⚠ Excel欠如 | ❌ 全欠如 | ⚠ テストゼロ | ⚠ 更新手順なし | ⚠ 希望登録なし | ✅ 基本対策済 | ⚠ 週40h未チェック |
   | ... | ... | ... | ... | ... | ... | ... | ... |

3. **共通発見の抽出**:
   - 3つ以上の視点から指摘された項目を「全視点共通の課題」として抽出
   - 複数視点から高評価を受けた項目を「強み」として抽出

4. **矛盾点の特定**:
   - 視点間で評価が正反対の項目を抽出（例: A視点では高評価、F視点では低評価）
   - トレードオフの候補として記録

### Phase 4: リリースブロッカーの特定

1. **ブロッカー候補の抽出**:
   以下の条件に該当する課題を「リリースブロッカー候補」として抽出:
   - **法令違反リスク**: 労働基準法違反、個人情報保護法違反、医療法違反など
   - **機能欠如**: REQUIREMENTS.mdに記載されているが未実装の重要機能
   - **複数視点から"Critical"指摘**: 3つ以上の視点から「致命的」と評価された課題
   - **実務投入不可要因**: 「これがないと現場で使えない」と明示された項目

2. **ブロッカーの構造化**:
   各ブロッカーを以下の形式で記述:

   ```markdown
   ### RB-X: ブロッカー名

   | 項目 | 内容 |
   |------|------|
   | **深刻度** | Critical |
   | **指摘視点** | A.管理者（主）, E.経営, G.法務 |
   | **内容** | 具体的な問題内容 |
   | **影響** | 経営影響、法的リスク、運用影響など |
   | **対応案** | 具体的な解決策 |
   ```

3. **影響度・緊急度のスコアリング**:
   - 影響度（High/Medium/Low）
   - 緊急度（Immediate/Short-term/Long-term）
   - リスク度（法的リスク、経営リスク、セキュリティリスク）

### Phase 5: 改善提案の統合

1. **全視点からの改善提案を集約**:
   - 各視点の「改善提案」セクションから提案をリストアップ
   - 重複する提案を統合（例: A視点とE視点が同じ「Excel出力」を提案している場合、1つにまとめる）

2. **優先度の再評価**:
   各提案を以下の基準で優先度付け:

   - **P0（リリースブロッカー）**: 法令違反/機能欠如で実務投入不可
   - **P1（リリース前推奨）**: 実務品質を担保するために強く推奨
   - **P2（リリース後早期）**: ユーザー満足度・保守性向上
   - **P3（中長期）**: 将来の拡張・スケーラビリティ

3. **工数の概算**:
   - 各提案の工数を「人日」単位で概算
   - 優先度別の合計工数を算出

4. **改善提案リストの作成**:

   | # | 提案 | 優先度 | 工数目安 | 担当視点 | 根拠 |
   |---|------|--------|---------|---------|------|
   | I-01 | 週40時間チェック制約の追加 | P0 | 2-3人日 | G, E | 労基法第32条違反回避 |
   | I-02 | 36協定対応（残業時間管理） | P0 | 5人日 | G, E | 月45h/年360hチェック |
   | ... | ... | ... | ... | ... | ... |

### Phase 6: トレードオフ分析

1. **トレードオフの特定**:
   Phase 3で抽出した「矛盾点」をもとに、以下のようなトレードオフを分析:
   - セキュリティ vs ユーザビリティ
   - 機能充実 vs リリーススピード
   - 暗号化 vs パフォーマンス
   - オフライン vs モバイル対応
   - 単一開発者 vs 保守性

2. **各トレードオフの構造化**:

   ```markdown
   ### T-X: トレードオフ名

   | 視点 | 主張 |
   |------|------|
   | F.セキュリティ | 認証・RBAC導入必須。DB暗号化必須 |
   | A.管理者 | ITリテラシーの低いユーザーが主体。認証は導入障壁 |
   | E.経営 | 「ダブルクリックで起動」が最大の売り。認証は障壁を上げる |

   **推奨判断**: ローカルデスクトップ用途では基本対策のみ実施。認証はスタッフ向けWeb機能追加時に導入。
   ```

3. **推奨判断の記述**:
   - 各トレードオフに対する推奨判断を記述
   - 判断の根拠を明記

### Phase 7: 殿の判断事項リストの作成

1. **判断事項の抽出**:
   以下の条件に該当する事項を「殿の判断事項」として抽出:
   - **戦略的判断**: リリース戦略、市場展開、技術選択
   - **コスト判断**: 投資判断、リソース配分、外部委託
   - **リスク判断**: リスク受容、リスク回避、リスク軽減策
   - **優先順位判断**: 複数のP0課題がある場合の優先順位
   - **トレードオフ判断**: 相反する要件のどちらを優先するか

2. **判断事項の構造化**:

   ```markdown
   ### 判断事項X: 判断事項名

   **選択肢**:

   | 選択肢 | 内容 | メリット | デメリット | 工数 | リスク |
   |--------|------|---------|-----------|------|--------|
   | A. ... | ... | ... | ... | ... | ... |
   | B. ... | ... | ... | ... | ... | ... |
   | C. ... | ... | ... | ... | ... | ... |
   ```

3. **推奨案の記述**:
   - 家老・将軍の立場からの推奨案を記述（ただし最終判断は殿が行う）

### Phase 8: 統合レポートの出力

1. **エグゼクティブサマリの作成**:
   - プロジェクトの概要（3つの差別化ポイントなど）
   - 致命的欠陥の要約（3〜5件）
   - ポテンシャル評価（「MVPになるか？」）

2. **統合レポートの構成**:

   ```markdown
   # {project_name} 多角的評価 最終統合レポート

   ## 1. エグゼクティブサマリ

   ## 2. 横断分析テーブル（N視点×評価軸マトリクス）

   ## 3. 致命的課題（リリースブロッカー）

   ## 4. 未確定要件リスト

   ## 5. 見直し推奨リスト

   ## 6. 改善提案リスト（優先度付き）

   ### 工数サマリ

   | 優先度 | 合計工数 | 項目数 |
   |--------|---------|--------|
   | P0 | XX人日 | X件 |
   | P1 | XX人日 | X件 |
   | P2 | XX人日 | X件 |
   | P3 | XX人日 | X件 |

   ## 7. 視点間の矛盾・トレードオフ

   ## 8. 全視点共通で指摘された問題

   ## 9. 次のアクション（殿が判断すべき事項）

   ## 付録: 各視点レポート参照先
   ```

3. **ファイル出力**:
   - 統合レポートを `output_path` に出力
   - Markdown形式（GitHub Flavored Markdown）

## Input Parameters

スキル実行時に以下のパラメータを受け取る：

- **`report_files`** (必須): 視点別評価レポートのファイルパスリスト
  - 例: `["queue/reports/shift_eval_A_manager.md", "queue/reports/shift_eval_B_staff.md", ...]`
  - 最低3つ以上のレポートが必要
- **`project_name`** (必須): 評価対象プロジェクトの名前
  - 例: `"shift-scheduler-claude"`
- **`target_version`** (任意): 評価対象のバージョン
  - 例: `"v1.5.0"`
  - デフォルト: `"latest"`
- **`output_path`** (必須): 統合レポートの出力先パス
  - 例: `"queue/reports/final_integrated_report.md"`

## Output Format

出力ファイルは以下の形式：

```markdown
# {project_name} 多角的評価 最終統合レポート

**作成日**: YYYY-MM-DD
**作成者**: ashigaru{N}
**対象**: {project_name} {target_version}
**入力レポート**: N視点（A.管理者 / B.スタッフ / ...）

---

## 1. エグゼクティブサマリ

{project_name}は、**差別化要素1、差別化要素2、差別化要素3**という3つの差別化を持つ{プロジェクト概要}である。{高評価された点}は高く評価された（{根拠}）。

しかし、**N個の致命的欠陥**がリリースを阻んでいる：

1. **欠陥1**（{視点}: {スコア}）― {詳細}
2. **欠陥2**（{視点}）― {詳細}
3. **欠陥3**（{視点}: リスク度 高）― {詳細}

これらN点は**全てリリースブロッカー**であり、解消なくして実務投入は不可能である。一方で、{高評価領域}は高い完成度を示しており、上記課題を解消すれば**実務投入可能なMVP**となるポテンシャルは十分にある。

---

## 2. 横断分析テーブル（N視点×評価軸マトリクス）

{マトリクステーブル}

---

## 3. 致命的課題（リリースブロッカー）

{各ブロッカーの詳細}

---

（以下、Phase 8で定義した構成に従う）
```

## Tips

### 1. 視点数と分析の深さ

- **3〜5視点**: コンパクトな分析。トレードオフは少なめ
- **6〜8視点**: 包括的な分析。トレードオフが多く出現
- **9視点以上**: 非常に詳細な分析。マトリクスが巨大化するため、評価軸を絞る必要がある

### 2. 視点の選び方

典型的な視点の組み合わせ：

- **B2C製品**: A.エンドユーザー、B.管理者、C.開発者、D.運用、E.経営、F.セキュリティ、G.法務
- **B2B製品**: A.購買担当、B.利用部門、C.IT部門、D.経営層、E.セキュリティ監査、F.法務・コンプラ
- **社内システム**: A.利用部門、B.IT部門、C.経営層、D.監査部門
- **OSS/技術基盤**: A.開発者、B.アーキテクト、C.運用、D.セキュリティ、E.コミュニティ

### 3. リリースブロッカーの判定基準

以下のいずれかに該当する場合、リリースブロッカーとして扱う：

- 法令違反リスク（労働基準法、個人情報保護法、医療法など）
- REQUIREMENTS.md記載の必須機能が未実装
- 3つ以上の視点から「Critical」指摘
- 「これがないと実務で使えない」と明示された項目
- セキュリティ上の重大な欠陥（SQL injection、XSSなど）

### 4. トレードオフの可視化

トレードオフは「二者択一」ではなく、「どちらを優先するか」の判断材料として提示する。
推奨判断には以下を含める：

- どちらを優先するか
- なぜそう判断したか
- 優先しなかった方の要件をどうカバーするか

### 5. 殿の判断事項の抽出

家老・将軍だけでは決められない事項のみを「殿の判断事項」として抽出する。
以下は**殿の判断事項ではない**：

- 技術的な実装詳細（家老・足軽で決定可能）
- 明確な法令要件（法律に従うのみ）
- 既にREQUIREMENTS.mdに記載されている要件（既定路線）

## Quality Checklist

スキル実行前に以下を確認：

- [ ] 入力レポートは3つ以上あるか？
- [ ] 各レポートは「良い点」「課題」「改善提案」セクションを含むか？
- [ ] プロジェクト名とバージョンは明確か？
- [ ] 出力先パスは正しいか？

スキル実行後に以下を確認：

- [ ] 横断分析テーブルは全視点をカバーしているか？
- [ ] リリースブロッカーは全て「複数視点からの指摘」または「法令違反」に該当するか？
- [ ] 改善提案はP0〜P3の優先度が明確か？
- [ ] 工数概算は全ての提案に付与されているか？
- [ ] トレードオフ分析には「推奨判断」が含まれているか？
- [ ] 殿の判断事項は「選択肢」形式で提示されているか？
- [ ] エグゼクティブサマリは3〜5行で要約されているか？

## Related Skills

- **個別視点評価スキル** (前工程):
  - このスキルは「視点別評価が完了している」ことを前提とする
  - 各視点の評価レポートを作成する専用スキルが別途必要（例: `stakeholder-evaluation-generator`）

- **ADR作成スキル** (後工程):
  - このスキルで抽出した「殿の判断事項」をもとに、ADR（Architecture Decision Record）を作成する
  - 関連スキル: `shogun-adr-creator`

- **スキル化候補抽出スキル**:
  - このスキルの実行結果自体が「スキル化候補」となる場合がある
  - 汎用的なワークフローを発見した場合は `skill-creator` で新規スキル化を検討

## Known Limitations

- **視点の粒度**: 視点が多すぎる（10視点以上）とマトリクスが読みづらくなる。視点を統合するか、評価軸を絞る必要がある
- **レポート品質依存**: 入力レポートの品質が低い（根拠がない、評価基準が曖昧）場合、統合レポートも品質が低下する
- **トレードオフの複雑性**: 3者以上のトレードオフ（A vs B vs C）は可視化が困難。二者択一に分解する必要がある
- **工数概算の精度**: 工数は「概算」であり、実際の開発では±50%のブレが発生する可能性がある
