# Community Research Report

## Overview

特定の技術（フレームワーク、ライブラリ、ツール等）について、コミュニティの声・ユーザー評価・市場動向を複数のプラットフォームから網羅的に収集し、体系的なMarkdownレポートとして出力するスキル。

**収集対象プラットフォーム:**
- 日本語圏: Qiita, Zenn, はてなブログ, note
- 英語圏: Stack Overflow, dev.to, Medium
- コード共有: GitHub (Stars, Forks, Issues, Contributors)
- 求人: レバテック, Indeed, フリーランスHub 等
- セキュリティ: NVD, JVN, Security NEXT

**用途:**
- 新技術の採用判断の材料収集
- 競合技術との比較分析
- 技術戦略策定のためのエビデンス収集
- RAGナレッジベースのコミュニティ情報パート

## When to Use

以下のいずれかに該当する場合にこのスキルを使用する：

- 「○○のコミュニティ情報を調査して」
- 「○○の評判・口コミを集めて」
- 「○○の良い点・課題点をまとめて」
- 「○○ vs △△の比較情報を集めて」
- 「○○の導入企業・業界傾向を調べて」
- 「○○の求人市場を分析して」
- 「○○のGitHub活動状況を調べて」
- 「○○のセキュリティ・脆弱性情報を確認して」
- 特定技術のコミュニティ規模・活発度を把握したい場合
- 技術採用判断のために多角的な情報収集が必要な場合

**トリガーキーワード**: コミュニティ調査, 評判調査, ユーザー情報, 市場動向, 技術比較, 求人分析, GitHub分析, 導入事例

## Instructions

### Phase 1: 技術ブログ調査

日本語・英語の技術ブログプラットフォームから対象技術に関する記事を網羅的に収集する。

#### Step 1.1: Qiita 調査（必須）

```
【検索パターン — 並列実行可能】

1. メイン検索
   WebSearch: "site:qiita.com {技術名}"

2. 最新記事検索（深度 standard 以上）
   WebSearch: "site:qiita.com {技術名} 2025 2026"

3. 別名・カタカナ検索（ある場合）
   WebSearch: "site:qiita.com {別名/カタカナ表記}"
   例: 「ナブラーク」「なぶらーく」

4. 関連キーワード展開（深度 thorough のみ）
   WebSearch: "site:qiita.com {技術名} {関連語}"
   展開語例: ハンドラ, 設定, トラブル, 入門, 比較

【抽出項目 — 各記事について】
- タイトル
- URL
- 投稿日（更新日）
- カテゴリ分類（入門/Tips/トラブルシューティング/レビュー/新機能 等）
- いいね数（WebFetchで確認可能な場合）
- 概要（2-3文）
```

#### Step 1.2: Zenn 調査

```
【検索パターン】
WebSearch: "site:zenn.dev {技術名}"

【実践知見】
- Zennはモダンな技術スタックの記事が多い
- エンタープライズ系FWの記事は非常に少ない傾向
- 記事が少ない場合、そのこと自体が重要な情報（コミュニティ特性の指標）
- 結果が0件でも「Zenn上にはほぼ存在しない」と明記する
```

#### Step 1.3: その他日本語ブログ

```
【検索パターン — 並列実行可能】

1. はてなブログ
   WebSearch: "{技術名} 所感 OR 使ってみた OR 感想"

2. 技術評論・企業ブログ
   WebSearch: "{技術名}とは 概要 特長 OR 導入"

3. note
   WebSearch: "site:note.com {技術名}"

【実践知見 — subtask_020で有効だったパターン】
- 「{技術名} 使ってみた」は個人の率直な評価が見つかりやすい
- 「{技術名}とは」は企業の解説記事（一創のようなSIer系）がヒットしやすい
- 「{技術名} 所感」は技術的に深い考察記事が見つかりやすい
- はてなブログ/noteは直接的なNablarch記事が少なかった（ニッチ技術の場合）
```

#### Step 1.4: 英語圏ブログ（深度 standard 以上）

```
【検索パターン — 並列実行可能】

1. Stack Overflow
   WebSearch: "{技術名} site:stackoverflow.com"

2. dev.to / Medium
   WebSearch: "{技術名} blog review tutorial"

3. Reddit
   WebSearch: "{技術名} site:reddit.com"

【実践知見】
- 日本発の技術は英語圏の情報がほぼゼロの場合がある
- その場合「英語圏コミュニティは事実上存在しない」と明記する
- グローバル技術（React, Terraform等）は英語圏の方が情報量が圧倒的に多い
```

### Phase 2: GitHub 分析

対象技術のGitHub上の活動状況を定量的に分析する。

#### Step 2.1: リポジトリ統計の取得

```
【GitHub CLI を使用 — 並列実行可能】

1. メインリポジトリの統計
   gh api repos/{owner}/{repo} --jq '{
     stars: .stargazers_count,
     forks: .forks_count,
     watchers: .watchers_count,
     open_issues: .open_issues_count,
     language: .language,
     license: .license.spdx_id,
     created: .created_at,
     updated: .updated_at,
     description: .description
   }'

2. オーガニゼーションの統計（存在する場合）
   gh api orgs/{org} --jq '{
     public_repos: .public_repos,
     description: .description,
     created: .created_at
   }'

【実践知見 — Nablarch調査の実績】
- nablarch/nablarch: Stars 42, Forks 11, 114 repos（2026-02時点）
- 比較基準: Spring Boot ≈ 76k stars, Quarkus ≈ 14k stars
- Stars数は技術のコミュニティ規模の良い指標
```

#### Step 2.2: コントリビューター・Issue活動（深度 thorough のみ）

```
【追加取得項目】
- 直近1年のコミット数
- Issue/PR のオープン・クローズ率
- コントリビューター数
- リリース頻度

gh api repos/{owner}/{repo}/contributors --jq 'length'
gh api repos/{owner}/{repo}/releases --jq '.[0:5] | .[] | {tag: .tag_name, date: .published_at}'
```

#### Step 2.3: コミュニティ特性の評価

```
【評価基準】
- 企業主導OSS: 公開メンバー非表示、外部PRが少ない
  → Nablarchはこのパターン。開発チームが東京・大阪に分散
- コミュニティドリブンOSS: 多数の外部コントリビューター、活発なIssue/PR
  → React, Vue.js等はこのパターン
- 個人プロジェクト: Stars多いがメンテナ1-2名
```

### Phase 3: 開発者評価の収集

ブログ記事やコミュニティの声から、対象技術に対する評価を構造化して収集する。

#### Step 3.1: 良い点/課題点の抽出

```
【検索パターン — 並列実行可能】

1. メリット・良い点
   WebSearch: "{技術名} メリット OR 良い点 OR 利点"

2. デメリット・課題
   WebSearch: "{技術名} デメリット OR 課題 OR 問題点 OR 難しい"

3. 学習コスト・ドキュメント品質
   WebSearch: "{技術名} 学習コスト OR ドキュメント OR 難しい"

【重要な記事は WebFetch で深掘り】
- いいね数が多い記事
- 率直な評価を含む「使ってみた」系記事
- 技術的に深い考察記事

【抽出フォーマット】
各評価について以下を記録:
- 評価内容（引用 or 要約）
- 出典URL
- 投稿日
- カテゴリ（良い点 / 課題点）
```

#### Step 3.2: 競合技術との比較コメント

```
【検索パターン】
WebSearch: "{技術名} vs {競合技術} 比較"
WebSearch: "{技術名} OR {競合技術} 選定 OR 比較 OR 違い"

【主な比較軸 — テンプレート】
| 観点 | {技術名} | {競合技術} |
|------|---------|-----------|
| 対象市場 | | |
| コミュニティ規模 | | |
| 学習コスト | | |
| クラウド対応 | | |
| エコシステム | | |
| ドキュメント | | |
| 求人数 | | |
| 最新トレンド追随 | | |

【実践知見 — Nablarch vs Spring の場合】
- TIS自身がSpringとNablarchの両方を活用する立場
- Nablarch: 日本市場特化、開発標準一式、ウォーターフォール向き
- Spring: グローバル、クラウドネイティブ、アジャイル対応
- 比較は9観点で対照表を作成した
```

### Phase 4: 市場動向

求人情報と導入企業情報から、対象技術の市場での位置づけを分析する。

#### Step 4.1: 求人情報の調査

```
【検索パターン — 並列実行可能】

1. 一般求人
   WebSearch: "{技術名} 求人 エンジニア"

2. フリーランス案件
   WebSearch: "{技術名} 案件 フリーランス 単価"

3. 転職市場
   WebSearch: "{技術名} 転職 年収 経験"

【収集項目】
- 求人数（概算）
- 平均単価・年収（確認可能な場合）
- 求められるスキルセット
- 求人が少ない場合はその理由を分析

【実践知見 — Nablarch調査の実績】
- Nablarch名指しの公開求人はほぼ0件だった
- ニッチ技術の場合、「Java SIer」等の上位カテゴリで検索し関連性を推測
- 求人が少ないこと自体が重要な情報（キャリアリスク指標）
- レバテックでの関連カテゴリ（Java: 3,496件、Spring: 1,130件）との対比が有効
```

#### Step 4.2: 導入企業・業界傾向

```
【検索パターン】
WebSearch: "{技術名} 導入企業 事例 採用"
WebSearch: "{技術名} {開発元企業} 金融 公共 製造 実績"

【収集項目】
- 導入業界（金融、公共、製造、流通等）
- 案件の規模・特徴
- 開発元企業以外での採用有無
- 公開されている導入メリット

【注意】
- 個人を特定する情報は記載しない（企業名のみ）
- 公開情報のみ収集（推測は推測と明記）
```

#### Step 4.3: ライセンス・課金情報（OSS技術の場合）

```
【確認項目】
- OSSライセンスの種類
- 有償サポートの有無・体系
- エンタープライズ版の有無
- 課金モデル（CPU コア数ベース、ユーザー数ベース等）
```

### Phase 5: セキュリティ・信頼性

対象技術の脆弱性情報とセキュリティ対応体制を調査する。

#### Step 5.1: CVE情報の確認

```
【検索パターン】
WebSearch: "{技術名} 脆弱性 セキュリティ CVE"
WebSearch: "{技術名} vulnerability CVE"

【収集項目 — 各CVEについて】
| CVE ID | 種類 | CVSS | 影響 | 修正版 | 発見日 |
|--------|------|------|------|--------|--------|

【実践知見 — Nablarch調査の実績】
- CVE-2019-5918 (XXE, CVSS 8.5「危険」) と CVE-2019-5919 (暗号化不備) を発見
- Security NEXT の記事をWebFetchで1ページ目・2ページ目の両方を確認
- JVN (Japan Vulnerability Notes) も有効な情報源
```

#### Step 5.2: セキュリティ対応体制の評価

```
【確認項目】
- JPCERT/CC との連携有無
- セキュリティアドバイザリの公開方法
- 脆弱性報告窓口の有無
- セキュリティ機能の標準搭載状況
```

### Phase 6: レポート生成

収集した全情報を体系的なMarkdownレポートにまとめる。

#### Step 6.1: 全体構成

「Output Format」セクションのテンプレートに従い、以下の構成でレポートを作成する。

```
1.  技術ブログ記事一覧（Qiita/Zenn/その他）
2.  GitHub・OSSコミュニティ活動
3.  開発者の評価・フィードバック（良い点/課題点）
4.  競合技術との比較
5.  導入企業・業界の傾向
6.  求人情報からの傾向分析
7.  セキュリティ・脆弱性情報
8.  公式コンテンツ・学習リソース
9.  良い点/課題点サマリテーブル
10. ソースURL一覧
```

#### Step 6.2: サマリテーブルの作成（必須）

```
【良い点サマリ】
| # | カテゴリ | 良い点 | 信頼度 |
|---|---------|--------|--------|
| 1 | ... | ... | 高/中/低 |

信頼度の基準:
- 高: 公式情報、複数ソースで裏付けあり
- 中: 個人ブログだが具体的なエビデンスあり
- 低: 単一ソース、主観的意見

【課題点サマリ】
| # | カテゴリ | 課題点 | 深刻度 |
|---|---------|--------|--------|
| 1 | ... | ... | 高/中/低 |

深刻度の基準:
- 高: 複数のユーザーが指摘、採用判断に直接影響
- 中: 一部ユーザーが指摘、回避策あり
- 低: 軽微な不満、限定的な影響
```

## Input Format

スキル実行時に以下のパラメータを指定する。

```yaml
# 必須パラメータ
technology_name: "Nablarch"                    # 対象技術名
output_path: "output/nablarch_kb_community.md" # 出力ファイルパス

# オプションパラメータ
depth: "standard"              # quick / standard / thorough（デフォルト: standard）
alternative_names:             # 別名・カタカナ表記（検索精度向上）
  - "ナブラーク"
  - "なぶらーく"
competitor_technologies:       # 比較対象技術（比較表の作成に使用）
  - "Spring Framework"
github_repos:                  # GitHubリポジトリ（統計取得用）
  - owner: "nablarch"
    repo: "nablarch"
  - org: "nablarch"            # オーガニゼーション
additional_search_terms:       # 追加検索キーワード
  - "Nablarch 導入"
  - "Nablarch メリット"
exclude_keywords:              # 除外キーワード
  - "Xenlon"
language: "ja"                 # 出力言語（デフォルト: ja）
```

### パラメータ説明

| パラメータ | 必須 | デフォルト | 説明 |
|----------|------|----------|------|
| `technology_name` | ✅ | — | 対象技術の名称 |
| `output_path` | ✅ | — | 出力Markdownファイルのパス |
| `depth` | — | `standard` | 調査深度。`quick`=主要プラットフォームのみ、`standard`=全プラットフォーム+GitHub、`thorough`=全プラットフォーム+GitHub深掘り+全記事WebFetch |
| `alternative_names` | — | `[]` | 別名・カタカナ表記 |
| `competitor_technologies` | — | `[]` | 比較対象技術 |
| `github_repos` | — | `[]` | GitHubリポジトリ・オーガニゼーション |
| `additional_search_terms` | — | `[]` | 追加検索キーワード |
| `exclude_keywords` | — | `[]` | 除外キーワード |
| `language` | — | `ja` | 出力言語 |

### 深度別の処理範囲

| Phase | quick | standard | thorough |
|-------|-------|----------|----------|
| 1. 技術ブログ | Qiitaのみ | Qiita+Zenn+ブログ+英語 | 全て+関連KW展開 |
| 2. GitHub | なし | 基本統計のみ | 統計+Issues+Contributors |
| 3. 開発者評価 | 検索結果の要約のみ | 主要記事WebFetch | 全記事WebFetch |
| 4. 市場動向 | なし | 求人+導入企業 | 求人+導入+課金体系 |
| 5. セキュリティ | なし | CVE検索のみ | CVE+対応体制+詳細分析 |
| 6. レポート | 簡易版 | 標準版（10セクション） | 完全版（全セクション+付録） |

## Guidelines

### 必須ルール

1. **公平性の確保**
   - 批判的な意見もポジティブな意見も同等に収集する
   - 「課題点を多く見つけた」場合でもフィルタリングしない
   - 「良い点しかない」場合はバイアスを疑い追加検索する

2. **情報の鮮度を記録する**
   - 全ての記事・情報に投稿日/更新日を付記する
   - 古い情報は「{YYYY年}時点の情報」と明記する
   - 最新版で改善されている可能性がある場合はその旨を注記する

3. **ソースURLを必ず付記する**
   - 全ての情報にソースURLを付ける
   - 「～という声がある」のような曖昧な記述は避け、具体的な出典を示す

4. **個人情報を記載しない**
   - 個人名は記載しない（企業名のみ）
   - Qiita等のユーザー名は内部分析用に使用するが、レポートには含めない

5. **並列実行を最大限活用する**
   - 独立したWebSearchは全て並列実行
   - 独立したWebFetchは4件まで並列実行
   - gh api呼び出しも並列実行可能

6. **「見つからなかった」も記録する**
   - 「Zenn上にはほぼ存在しない」→ コミュニティ特性の指標
   - 「求人が0件」→ キャリアリスク指標
   - 「英語圏の情報がない」→ グローバル展開の限界
   - 情報の不在そのものが重要な発見である

### 検索のコツ（実践知見）

1. **キーワード展開が発見率を大幅に向上させる**
   - メインキーワードだけでは60-70%の記事しか見つからない
   - 別名、カタカナ、関連語で追加検索すると90%以上をカバー

2. **「{技術名} 使ってみた」は最も率直な評価が得られるパターン**
   - 個人開発者の率直な感想が見つかりやすい
   - 良い点・悪い点の両方が含まれることが多い

3. **「{技術名}とは」は企業系解説記事のパターン**
   - SIer系企業のマーケティング記事が多い
   - 導入メリットは詳しいが、課題点は控えめなバイアスに注意

4. **年号を含めた検索で最新記事を優先発見できる**
   - 「{技術名} 2025 2026」で直近の記事を効率的に発見

5. **WebFetchは主要記事に集中する**
   - いいね数が多い記事、率直な評価を含む記事を優先
   - 入門チュートリアルはタイトル・概要のみで十分

### アンチパターン（避けるべきこと）

- **ポジティブ偏向**: 良い点ばかり集めて課題を無視する
- **古い情報の過大評価**: 5年以上前の記事を現在の評価として引用
- **単一ソースへの依存**: 1つの記事だけで「○○と評価されている」と断定
- **コミュニティ不在の無視**: 「情報が見つからない」をレポートに含めない
- **推測と事実の混同**: 推測は「推測される」「可能性がある」と明記
- **逐次的な検索実行**: 並列可能な検索を1つずつ実行する

### 深度別の目安

| 深度 | WebSearch | WebFetch | gh api | レポート行数 |
|------|-----------|----------|--------|------------|
| quick | 5-8回 | 3-5件 | 0回 | 100-200行 |
| standard | 12-18回 | 8-15件 | 2-4回 | 300-600行 |
| thorough | 20-30回 | 15件以上 | 5-10回 | 500行以上 |
